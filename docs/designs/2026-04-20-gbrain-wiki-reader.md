## Vision

### 10x Check

The design doc as written is a beautiful reader. Ships tonight. That's v0 Approach B.

The 10x version, pulled from the user's own office-hours insight: **Willipedia is the input layer, not the output layer.** Not a reader for GBrain — a continuous feedback pipe from how the user actually uses their brain back *into* GBrain. Every hover, every empty-result search, every broken wikilink followed, every page re-read, every scroll-past is signal. The reader does the work of telling GBrain what the user doesn't know yet. GBrain does the work of filling it. Over weeks, the brain literally adapts to the user's attention.

### Platonic Ideal (12 months)

The user opens Willipedia and the home page says: *"You've searched for 'AI agent orchestration' three times this week and I don't have a strong page on it. I queued it for expansion two hours ago — want to see the draft?"* Or: *"You've visited the Karpathy page four times but never clicked the three outgoing transformer links. Want those in your queue?"*

The reader stops being a document browser. It starts being a collaborator that notices patterns and shapes GBrain accordingly. Reading becomes bidirectional.

**Note on v0 scope after outside-voice trim:** the platonic ideal is directional, not load-bearing for tonight's build. Telemetry tables still land (so v1 has data), but the `/stats` dashboard and Service Worker are deferred. v0 remains a beautiful reader that quietly records signal; v1 consumes and visualizes it.

## Acceptance Criteria (per accepted expansion)

- **Telemetry:** Three tables written on boot if migrations run cleanly. Page view insert fires inside every `/page/*` handler (after render, fire-and-forget). Broken-link insert fires in the wikilink rewriter when target `.md` is absent. Search insert fires inside `/api/search`. **Stderr logging is throttled to once per minute per error type** (not per failed insert) — a flapping Postgres must not spam logs. **Migration failure is a degraded-mode trigger, not a crash:** if migrations fail at boot, telemetry inserts become no-ops, smart home collapses to "Recently updated" only, and one stderr line per minute reports the state. Acceptance: visit three pages, open Postgres, see rows. Kill Postgres, reload a page — rendering works, one (not many) stderr warning per minute. Kill Postgres before boot — migrations fail, app still serves pages, smart home shows only "Recently updated".
- **Backlinks panel:** Every `/page/*` render includes a `<BacklinksPanel slug={slug} />` island that fetches `/api/backlinks/[slug]`. Empty state: "No pages link here yet." Loading: skeleton rows. Error (wrapper down): no panel + one stderr line (throttled, same rule as telemetry). Acceptance: visit a page with N backlinks, see N entries grouped by parent folder.
- **Cmd+K:** Global keydown listener on window. Modal centered, focus-trapped, closes on Escape. Empty input shows last 10 visited slugs from localStorage. Typed input debounces 150ms then calls `/api/search`. Arrow keys move selection, Enter navigates, Cmd+Enter opens new tab. Acceptance: works on `/`, `/page/*`, `/search` — no route where it fails.
- **Scroll memory + breadcrumb:** Scroll position restored **on first stable layout** (after `document.fonts.ready` + one `requestAnimationFrame`) if saved position exists and is <24h old. Clamp restore to `scrollHeight` in case page shrank. Breadcrumb state lives in `sessionStorage` keyed by tab session so it survives Astro's full-page MPA reloads; appears in the header after 2+ page visits in the session. Acceptance: load a long page, scroll 30%, navigate away, return — position restored after fonts load (no flash of top-of-page). Click three wikilinks in succession — breadcrumb shows all three even though each nav is a full reload.
- **Smart home (simplified):** Home route queries three sources with independent failure isolation. "Continue reading" = `SELECT slug FROM page_views ORDER BY ts DESC LIMIT 5` with dedupe. "Recently updated" = filesystem mtime over $GBRAIN_PATH. "Haven't visited in a while" = `SELECT slug, MAX(ts) FROM page_views GROUP BY slug ORDER BY MAX(ts) ASC LIMIT 5`. If telemetry has zero rows OR migrations failed, sections #1 and #3 hide; section #2 renders alone. Acceptance: fresh install → only "Recently updated" visible. After reading three pages → "Continue reading" appears. Kill Postgres → fresh-install behavior.
- **Print:** `Cmd+P` on any `/page/*` produces a clean document: serif body, no sidebars/TOC/search/header, slug + date footer, proper page breaks at headings. Acceptance: print-preview looks like an article, not a web page.
- **Test pyramid:** (1) Unit — `bun test`, covers wikilink rewriter (input MD → output HTML with rewritten links + broken-link telemetry captured), session-ID rotation (TTL logic), scroll-clamp helper. (2) Integration — spins up a test Postgres (docker or in-repo schema reset), asserts the three insert paths write rows; MCP client is tested against a mocked wrapper (no real bearer). (3) E2E — Playwright, two specs: happy path (home → search → page → wikilink → back) and Cmd+K (open via shortcut → type → navigate). Acceptance: `bun test` green, `bun run e2e` green in CI-ish local mode.

## Cross-Decision Architecture Impact

The original design doc specified **no Willipedia database**. Expansion 2 reverses that. Cascade:

- `migrations/` no longer deletable — Willipedia has its own tables (tentatively `page_views`, `search_log`, `broken_links`).
- `src/lib/db.ts` is added back (fresh file, not the old Postgres/pgvector module).
- `.env.example` adds `WILLIPEDIA_DATABASE_URL`.
- Migrations run on app boot. **On failure: log ONE stderr line, mark the DB as "degraded" in a module-level flag, and serve all pages without telemetry.** Rendering must not block on migrations. Every feature that depends on telemetry (#6 smart-home sections 1 and 3) must check the degraded flag and render its empty state.
- Deploy surface adds Postgres dependency. **Open sub-decision for eng review:** same Postgres instance as the MCP wrapper (separate schema `willipedia`), or a dedicated DB? Either works; recommend same-instance separate-schema for ops simplicity.
- Stderr noise from fire-and-forget inserts is throttled to once per minute per error-type (module-level Map<errorCode, lastLoggedAt>).
- **Test surface:** integration tests need a writable Postgres. Use a throwaway schema (`willipedia_test`) with migrations run in a `beforeAll`. E2E tests run against a real booted dev server + mocked wrapper so E2E stays deterministic.

## Accepted Scope (added to v0)

- **Postgres + telemetry tables:** `page_views(ts, slug, scroll_depth, session_id)`, `search_log(ts, query, results_count, session_id)`, `broken_links(ts, from_slug, target_slug)`. All fire-and-forget. **Session ID is a random opaque token in localStorage, persisted for 24h of inactivity (not wall-clock daily rotation). Any page view or search extends the TTL.** This keeps a single subway reading session from straddling two session IDs across rotation boundaries while still rotating often enough for trend analysis.
- **Backlinks panel:** right-column widget below TOC on desktop, below article on mobile. Client-side fetch to `/api/backlinks/[slug]`. Grouped by parent folder.
- **Cmd+K command palette:** Astro island bound to Cmd+K globally. Centered modal. Live-updating search results. "Recently viewed" when input empty.
- **Reading position memory:** `localStorage['readPosition:{slug}'] = scrollY` throttled to 2s; restore on load if fresh (<24h), after fonts are ready and layout is stable. Clamp restore to `scrollHeight`.
- **Session breadcrumb:** slug stack in `sessionStorage` keyed by tab session. Survives full-page MPA reloads within the same tab. Clears when tab is closed. Renders as subtle header breadcrumbs after the 2nd page visit in the session.
- **Smart home page:** three sections (Continue reading / Recently updated / Haven't visited). First-run fallback = just Recently updated. "Haven't visited" is a simple oldest-last-visited sort, NOT backlink-weighted.
- **Print stylesheet:** `@media print` in global.css. Hides header/TOC/sidebars. Adjusts margins/font/orphans. Subtle footer with slug + date. "Print this page" link in article header.
- **Test pyramid:** Unit + integration + Playwright E2E (happy path + Cmd+K only — SW E2E dropped with #7). `bun test` and `bun run e2e` scripts in `package.json`.

## Eng Review Handoff Notes

Questions the eng review must answer before implementation begins:

1. **Postgres topology** — wrapper's instance + `willipedia` schema, or fresh DB? Recommend same-instance for ops simplicity.
2. **Postgres client** — `postgres` (postgres.js), `pg`, or Bun's native SQL? Bun SQL is appealing given the Bun-everywhere stack.
3. **Session ID strategy** — anonymous opaque token with 24h-inactivity TTL? Reset on bearer change? Affects the telemetry schema.
4. **Cmd+K on touch** — it's global. What's the mobile behavior? Tap a visible search button opens the same modal.
5. **Backlinks API shape** — confirm wrapper exposes `list_backlinks` with the fields we need (`slug`, `title`, `parent_folder`). Captured in design doc Open Question #3. **Fallback if absent:** ship v0 without backlinks and move it to TODOS.md.
6. **Scroll restore edge cases** — restore after `document.fonts.ready` + one rAF. Clamp to `scrollHeight`. If GBrain rewrote the page mid-session, saved position is stale but bounded.
7. **Telemetry PII** — the `search_log` table contains raw query text. Single-user single-tenant, so technically fine, but worth a conscious "yes, I want to keep raw queries" decision rather than silent acceptance.
8. **Degraded-mode matrix** — spec every feature's behavior when telemetry is unavailable (migrations failed OR Postgres down post-boot). Baseline: smart home collapses to Recently Updated, Cmd+K recent-viewed falls back to localStorage, breadcrumb unaffected (sessionStorage), backlinks unaffected (wrapper call).
9. **Stderr throttle implementation** — module-level Map<errorCode, timestamp>; first occurrence logs, subsequent occurrences within 60s are silent. On flush-interval elapsing, log one summary line.
10. **Playwright CI posture** — run E2E headed in dev, headless in CI. First decision: is there a CI at all for Willipedia v0, or is `bun run e2e` a local-only gate the user runs before `ship`?
11. **Test-DB strategy** — shared `willipedia_test` schema wiped before each run, or per-test transactions? Per-test transactions are faster and cleaner; requires all inserts to be transaction-aware.

### P1-2. Hover state on wikilinks
DESIGN.md:187 specifies the resting state (dotted oxblood underline offset 0.18em) but not the hover delta. Both outside voices flagged this.

**Spec:**
- Hover: underline-offset grows from 0.18em to 0.22em (subtle). Color unchanged. No weight change, no background, no cursor change beyond the default pointer.
- Focus-visible (keyboard tab): add a 1px oxblood outline offset 2px around the link text box. Not the same as hover — the keyboard user needs a stronger affordance.
- Active (click in flight): nothing. The page will navigate before the browser's default pressed-state matters.

### P1-3. Hover preview card (for wikilinks)
The plan references hover preview (`plan:188`, `TODOS.md:31`) but DESIGN.md doesn't spec the visual. At minimum the card needs a spec before any implementation attempt.

**Spec (hover-preview.v0):**
- 360px wide, anchored to the link's right edge with a 12px offset. Flips to left edge if it would exceed viewport.
- Paper fill (same `#F4EDE0`), 1px ink border at 40% opacity, 0 radius, 0 shadow.
- Inner padding 24px. Content: title (EB Garamond wght 500, 20px, ink) → 2-line dek (Fraunces 14px, ink) → small-caps muted metadata line `UPDATED · 3 DAYS AGO · 4 BACKLINKS`.
- Shows after 300ms hover delay (per DESIGN.md respect for reduced-motion — this delay is intentional). Hides immediately on mouseleave of link + card group.
- If `prefers-reduced-motion: reduce`: the card appears instantly with no fade.
- Loading state: border fades in at opacity 0.4, content replaced by three muted small-cap dots (`· · ·`). If preview fetch >500ms: card shows title only, no dek.
- Error state: card does not appear. Link behaves as a normal wikilink. No error UI for a mere hover miss.

### P1-4. Broken-wikilink inline affordance
DESIGN.md:190 describes the treatment (small-caps, muted, dotted underline) but stops short of inline interaction. Claude outside-voice flagged this: the 404 is a page, but the wikilink is inline — what happens on click?

**Spec:**
- Click navigates to `/page/{slug}` as normal. Server detects absent `.md` and returns the 404-nudge page.
- Hover preview card is NOT attempted for broken wikilinks (no content to preview). The hover shows a lightweight fragment instead: a tiny card with just a muted-italic line `No page yet. Click to propose one for research.` No loading state (there's nothing to load).
- In v0, clicking is the only affordance. The inline link doesn't sprout its own inline nudge button; the nudge lives on the 404 page where the card-catalog drawer illustration earns its space.

### P1-5. Backlinks gutter — missing states
DESIGN.md:171 specifies the 240px gutter but only for happy path. Codex flagged 0/1/20+ as implicit holes.

**Spec:**
- **0 backlinks:** gutter renders the header `BACKLINKS` in small-caps oxblood, then a single Fraunces italic muted line: `No pages link here yet.` Gutter width stays 240px — don't collapse; preserving the geometry matters more than saving pixels.
- **1 backlink:** same header, one entry, no plural. Metadata line below entry reads `1 incoming link`.
- **20+ backlinks:** paginate at 12. Footer shows `SHOW ALL 27 →` small-caps oxblood link that expands in place (no modal, no new route).
- **Loading (PR #2+ when backlinks become client-fetched):** skeleton rows — three muted hairline blocks at slightly varying widths. No shimmer, no pulse (that's the wrong vocabulary).
- **Wrapper down:** gutter renders header + one small-caps muted line `UNAVAILABLE` and a throttled stderr log line. No retry affordance (the retry happens on next page navigation).
- **Mobile:** backlinks collapse to a horizontal rule followed by `BACKLINKS` header and an inline list BELOW the article. See P1-6 for exact mobile geometry.

### P1-6. Responsive spec (what DESIGN.md leaves ambiguous)
DESIGN.md flags marginalia-mobile as open (line 251). My recommendation based on the broadsheet brief:

**Breakpoints:**
- Desktop >=1280px: 38% content-well offset, 680px body measure, 240px right marginalia gutter. Matches DESIGN.md.
- Tablet 768-1279px: content well centers (no left offset), 640px measure, marginalia still in right gutter but narrower (200px) with inline-below fallback if viewport would leave <540px for body.
- Mobile <768px: single column, 100% width minus 40px side padding, body measure becomes responsive (max 520px). **Marginalia: inline-below-paragraph, muted italic, indented 24px left, typographic rule above each marginalia note.** (This resolves DESIGN.md's open question — inline, not drawer. Drawer is a UI pattern; inline is a typography pattern, and we picked typography-first at every other fork.)

**Why inline not drawer:**
- A drawer is a screen affordance that doesn't map to a printed book. A broadsheet handed to a reader doesn't have drawers; it has footnotes and sidenotes.
- Inline keeps marginalia visible at read time, which is where the typography thesis pays off.
- Drawer adds JS + state + interaction cost for a gain (main-column cleanliness) that mobile readers are already used to losing.

### P1-7. Cell vs card nomenclature
`v0-home-fallback-approved.json:entry_card_spec` calls the home entries "cards." DESIGN.md:217 and :224 ban "cards" as a UI pattern.

**Resolution:** the home entries are **cells**, not cards. No border-radius, no fill, no shadow, no stacking. They are typographic cells in a grid separated by hairlines. Update `v0-home-fallback-approved.json` to rename `entry_card_spec` → `entry_cell_spec` and to add: `no border-radius, no fill, no shadow, no stack — cells are type-on-paper in a hairline grid`.

### P1-8. External-link glyph
DESIGN.md:194 names both pilcrow (¶) and section mark (§) as candidates. Codex flagged the undecided state.

**Recommendation: pilcrow (¶).** Reasons:
- Section mark (§) is legally-coded in reader culture (statute citations, journal articles). Willipedia is neither.
- Pilcrow is more typographically idiomatic for "this link leaves the paragraph/text flow" — exactly what an external link does.
- Matches the same-rule-everywhere discipline: pilcrow also happens to be a valid marginalia marker in longer-form typography, which keeps the visual vocabulary compact.

### P1-9. Cold-boot font-load degradation (Claude outside voice — surprising observation)
Fonts are Google Fonts in v0. If fonts fail to load (Tailscale flakes; Google's CDN has a brief outage; DNS drift), the page renders with system serif/sans fallbacks. That's not "degraded" — that's "the entire aesthetic thesis failed silently."

**Resolution (v0, tonight):**
- Add `<link rel="preload" as="font" crossorigin ...>` for the three most-critical faces: EB Garamond 500 (masthead), Fraunces 400 opsz 20 SOFT 50 (body), EB Garamond SC 400 (small caps).
- CSS `font-display: swap` — when fonts fail, body text still renders in a fallback, but the fallback stack is explicitly tuned: `'Fraunces', ui-serif, Georgia, 'Times New Roman', serif` — NEVER the default browser serif.
- Ship gate: cold-boot a fresh Safari tab with devtools throttling set to "Offline" for `fonts.gstatic.com` and verify the page is still readable. If the fallback stack renders wrong (e.g., masthead becomes Times), adjust the stack before shipping.
- PR #2+ queue: self-host the three faces. Google Fonts for a single-user app is an unnecessary CDN dependency.
